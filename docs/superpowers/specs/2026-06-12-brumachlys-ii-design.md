# BRUMACHLYS II — Build Specification

> *brume* (fr. mist) + *achlys* (gr. Ἀχλύς, the death-mist of the Iliad)
> Simultaneous-resolution tactical wargame on a procedurally generated organic board.
> Successor to the Brumachlys PoC (2026-05). Spec date: 2026-06-12.

---

## 0. TL;DR for implementing agents

Build a 2-faction wargame where the player and an AI commit orders blind each round, then a
deterministic resolver plays them out in initiative order. Combat math is the proven Weewar
formula (ported from PoC v1). **The board is NOT a hex grid**: it is an irregular cell graph
generated by an Oskar Stålberg-style relaxed quad mesh (ported from the `oskar-procedure`
project). Map shapes and terrain come from real Weewar map XMLs used as "donors".

**The /goal (v1 done means all of these):**
1. Playable solo vs AI on localhost (`npm run dev`).
2. Backed by a GitHub repo under `kai-denrei`.
3. Published as a static page (GitHub Pages), with cache-bust version badge visible.

**Hard rules:**
- **Determinism.** Same seed + same orders → byte-identical event logs. All randomness through
  the seeded RNG in `src/core/rng.ts`. `Math.random()`, `Date.now()`, argless `new Date()` are
  forbidden in `src/board/`, `src/core/`, `src/ai/`.
- **Purity.** `board/`, `core/`, `ai/` import nothing from `ui/`, `state/`, or the DOM.
- **Port, don't reinvent.** Two reference codebases are ours and quarryable:
  - `/Users/minikai/Documents/Dev/STB_Brumachlys` (v1) — combat, resolver structure, fog,
    Weewar XML parser, and 149 tests worth of vectors. Hex-specific code does NOT port.
  - `/Users/minikai/Dev/oskar-procedure` — mesh kernel (`src/grid.js`, `src/poisson.js`,
    `src/hex.js`, `src/halfedge.js`, `src/dual.js`, `src/rng.js`, `src/vec.js`) and its tests.
    Vanilla JS → translate to typed TS modules.
- **Decisions go to deban.** Any choice this spec leaves open: pick decisively, record it via
  the `.deban` decision log (this project uses deban instead of a DECISIONS.md).
- **Mobile-first.** Design for a phone in portrait. The board is the screen; chrome is minimal;
  everything secondary lives in modals/bottom sheets.
- Placeholder visuals are fine but must be *pleasant*: minimalist SVG unit glyphs, textured
  pastel cells. The render layer must make later sprite/animation swaps trivial (§10.4).

## 2. Game model

### 2.1 Round structure

```
┌────────────────────┐   ┌──────────────────────────┐   ┌────────────┐
│  PLANNING          │ → │  RESOLUTION              │ → │ END /      │
│  player queues     │   │  A: movement (init order)│   │ NEXT ROUND │
│  orders; AI plans  │   │  A.5: melee brawls       │   │            │
│  simultaneously    │   │  B: combat (init order)  │   │            │
└────────────────────┘   └──────────────────────────┘   └────────────┘
```

Solo play: no handoff screen. The AI's `planOrders()` is called when the player commits.

### 2.2 Initiative

Every unit type has `initiative: number`. During resolution, orders sort by initiative
descending; ties broken by FNV-1a 32-bit `hash(unitId + ":" + round)` ascending (port v1's
`rng.ts` FNV implementation). This is the only ordering mechanism inside a round.

### 2.3 Orders

```ts
type Order =
  | { kind: 'move';   unitId: string; path: CellId[] }   // destination cells, start excluded
  | { kind: 'attack'; unitId: string; targetCell: CellId }
  | { kind: 'stance'; unitId: string; stance: Stance };
```
Max one `move` + one `attack` + one `stance` per unit per round. Stance orders apply at
resolution start, before Phase A.

### 2.4 Stances

| Stance | Auto-attack (Phase B) | Counter-attack | Modifier |
|---|---|---|---|
| `aggressive` (default) | yes — nearest visible enemy in range | yes | — |
| `defensive` | no | yes | +1 armor (adds to `Td`) |
| `hold-fire` | no | **no** | — |

Explicit attack orders execute in any stance except `hold-fire` (UI blocks entering one).
Auto-attack target tie-break: lowest current `count`, then lowest `hash(unitId+":"+round)`.

### 2.5 Phase A — movement (init order)

- Each `move` order traverses its path step by step, spending terrain cost (tenths) from the
  unit's `movement` budget. Pathing legality is re-checked at execution time.
- **Friendly-occupied cell mid-path:** pass through allowed. May not END on a friendly cell:
  stop one cell back (if none, stay put).
- **Enemy-occupied cell mid-path:** stop in the previous cell (surprise contact).
- **Enemy-occupied final destination at execution time:** the move COMPLETES into the enemy's
  cell — a **charge**. Brawl follows in Phase A.5. (This holds even if the enemy was hidden by
  fog at planning time and even if the enemy arrived earlier this Phase A.)
- **Two units, same empty destination:** higher initiative claims it; the later mover stops one
  cell back if the earlier occupant is friendly, or charges in if enemy.

### 2.6 Phase A.5 — melee brawls

For every cell occupied by units of both factions after Phase A:
repeat `battleExchange(higherInit, lowerInit)` (full mutual exchange, `B = 0`, terrain bonuses
apply for both since they share the cell, stances ignored) until one side's count is 0. Both
reaching 0 in the same exchange is legal — cell ends empty. Survivor keeps the cell.

### 2.7 Phase B — combat (init order)

- Explicit `attack` orders fire if target still in range after movement; otherwise the attack
  **fizzles** (logged `lost-target`, no fallback).
- Aggressive units without an explicit attack auto-attack per §2.4.
- Damage applies immediately (a later ally hits a softened defender — concentrate-fire is
  intentional).
- Counter-attack: within the attacker's slot, if the attacker is inside the defender's range
  and the defender is not in `hold-fire`: defender returns fire with `B = 0`. Both damages are
  computed against pre-exchange counts, then applied together (one tick of mutual loss).
- A unit reaching count 0 is removed; its remaining queued orders drop.

### 2.8 Win condition

A faction with zero units at end of round loses. Both at zero → draw. Round 40 reached with
both alive → draw ("the mist settles"). Banner + New Battle options (§9.6).

## 3. Board model

### 3.1 Generation pipeline (port from oskar-procedure)

1. **Seed points** — Poisson disk (default) or hex-lattice variant. Seeded PRNG.
2. **Delaunay triangulate** — vendor `delaunator` (copy oskar-procedure's vendored ESM).
3. **Dissolve** — merge triangle pairs into quads (convexity + angle bounds [36°, 162°]).
4. **Subdivide** — every face → quads via centroid + edge midpoints (watertight).
5. **Relax** — closest-square fit, fixed iteration count (run to completion synchronously at
   generation time; no animation needed for the game).
6. **Dual extraction** — one cell per interior primary vertex: polygon of incident quad
   centroids ordered by angle. Adjacency: two dual cells are neighbors iff their primary
   vertices share a quad edge (derive from the half-edge structure).

### 3.2 The game-facing types

```ts
type CellId = number;            // stable: index in generation order
type Vec2 = [number, number];

type Cell = {
  id: CellId;
  center: Vec2;                  // primary vertex position — angles, distances, placement
  polygon: Vec2[];               // CCW render ring
  neighbors: CellId[];           // sorted ascending for determinism
  terrain: TerrainKey;
};

type Board = {
  cells: Map<CellId, Cell>;      // only playable cells (deleted cells absent)
  seed: number;
  donorMapId: string;
};
```

All game logic sees only `Board` — never the mesh. Geometry helpers in `src/board/geometry.ts`:
- `graphDistance(board, a, b): number` — BFS hops (range, vision).
- `angleAt(board, pivot, a, b): number` — degrees in [0, 180] between vectors
  `center(pivot)→center(a)` and `center(pivot)→center(b)`.
- `cellsWithin(board, from, hops): CellId[]`.

Cell size/shape is **purely visual**. Every cell is one game space.

### 3.3 Determinism

Board generation is a pure function `generateBoard(donorMap, seed, targetCells): Board`.
Same inputs → identical cell ids, polygons, adjacency. Tests assert this.

## 4. Map generation — Weewar donors

### 4.1 Donor pipeline

Input: a Weewar map XML (format documented in v1's `src/io/weewar-xml.ts` — port the parser,
keep its tests). Steps:

1. Parse → sparse tile list (terrain per (x,y)), bases, start units, `maxPlayers`.
2. Keep factions 0 and 1; drop others' units/bases (warn in console).
3. Compute hex-center positions for tiles (odd-r offset → pixel), normalize bounding box into
   the unit square with 5% margin.
4. `targetCells = clamp(donorTileCount, 60, 250)`. Generate mesh sized to produce ≈ that many
   dual cells.
5. **Terrain assignment:** each dual cell takes the terrain of the nearest donor tile center.
   Cells whose nearest tile is farther than 1.4× the donor tile pitch are **deleted** — the
   board silhouette follows the donor landmass.
6. **Connectivity guard:** keep only the largest connected component of non-water cells plus
   any water cells adjacent to it (lakes render, are impassable). If the component holds < 80%
   of land cells or cannot host both factions' placements, retry with seed+1 (max 8 retries,
   then error visibly).
7. **Force placement:** each faction's anchor = the cell nearest its first donor base (or its
   first start unit if no base). Place the faction's army (§6.4) on distinct passable land
   cells by ascending BFS distance from the anchor, deterministic order.

### 4.2 Bundled donors

Bundle ~5 curated XMLs in `data/maps/` (imported as raw strings via Vite `?raw`). Selection
criteria: `maxPlayers == 2`, 150–500 tiles, produces a connected interesting board (verify by
generating). The donor stash is `/Users/minikai/Documents/Dev/STB_Brumachlys/weewar-maps/`
(12k+ files — scan with a script, do not bundle the stash). Record chosen maps in deban.
`Tai Chi` (10701.xml) is a known-good 2-player candidate.

### 4.3 New battle flow

"New Battle" offers: same donor / new seed, or different donor. Seed shown and enterable
(shareable battles). Default seed source: a counter persisted in localStorage — **not**
`Date.now()` inside core; the UI layer may use `Date.now()` to pick a fresh seed, core may not.

## 5. Combat

### 5.1 Pluggable resolution

```ts
interface ResolutionModel {
  key: string;                                   // 'weewar' for v1
  attackDamage(ctx: AttackContext): number;      // pure
  battleExchange(ctx: ExchangeContext): ExchangeResult;
}
```
The resolver depends only on this interface. v1 of II ships `weewar`. Future models
(`simple`, `stance-rps`) are parking-lot game modes — the interface is the R&D enabler.

### 5.2 The weewar model (port from v1 `core/combat.ts`)

```
p = clamp(0.5 + 0.05 * ((A + Ta) - (D + Td) + B), 0, 1)
damage = round(attackerCount * p)        // Math.round, half-up — document
if damage == 0 and p > 0: damage = 1     // min-damage floor (v1 decision B.16)
```
- `A` = attacker's attack strength vs defender's armor type. `D` = defender armor.
- `Ta`/`Td` = terrain attack/armor bonuses (+1 `Td` if defender stance defensive).
- `attackerCount` = attacker's current sub-unit count (1–10).

### 5.3 Angle-based gang-up (the II innovation)

Each defender accumulates `attackedFrom: { cell: CellId; ranged: boolean }[]` within a round
(cleared at round end; counter-attacks do NOT append). At each attack, before damage:

For each prior entry, classify and sum into `B`:
- prior `ranged == true` (attacker was at graphDistance > 1 when it fired): **+1**
- else θ = `angleAt(board, defenderCell, priorEntry.cell, currentAttackerCell)`:
  - θ < 60° → adjacent **+1**
  - 60° ≤ θ < 135° → flanking **+2**
  - θ ≥ 135° → opposite **+3**

This reproduces hex semantics exactly (hex neighbors sit at 60° steps: adjacent 60°,
flanking 120°, opposite 180°) but works on any cell geometry. Test vectors in §13.3.

### 5.4 Reference combat vectors (must hold, ported from v1)

- Infantry(10) attacks Tank(10), both plains, B=0: A=3 (inf vs armored), D=5 →
  p = 0.40 → damage 4.
- Tank(10) attacks Infantry(10), plains, B=0: A=5 (tank vs personnel), D=6 →
  p = 0.45 → damage 5.
- Second attacker at θ ≥ 135° after the first: B=3 → p = 0.55; if tank count is now 6 →
  damage = round(6 × 0.55) = 3.

## 7. Fog of war

- `visibleCells(state, faction) = ⋃ cellsWithin(board, unit.cell, unit.vision)` over the
  faction's living units.
- **Terrain is always visible** (the map is known). Only enemy units hide.
- No last-known-position markers in v1 of II (parking lot).
- Planning: player sees own units + visible enemies. Move targets may be anywhere passable
  (terrain is known); attack targets must be visible and in range.
- **Replay fog (solo):** the replay renders through the PLAYER's fog, recomputed as units move
  during playback. AI units moving entirely outside player vision are not shown. Damage
  arriving from an unseen ranged attacker shows an impact marker on the defender with source
  hidden — "fire from the mist" (on-theme, keeps the AI's positions secret).
- The AI receives only its own fog-filtered view (§8.1) — symmetric honesty.

## 8. AI

### 8.1 Interface and fairness

```ts
interface OrderPlanner {
  key: string;
  planOrders(view: FactionView, rng: Rng): Order[];   // pure, deterministic
}
```
`FactionView` = game state filtered through the AI faction's fog: own units full detail,
enemy units only if visible, full terrain. Built by the same `visibleCells` used for the
player. The AI never reads hidden state. Swappable for stronger planners later.

### 8.2 v1 heuristic (`planner-greedy.ts`)

Per own unit, in initiative order:
1. Enumerate candidate destinations = reachable cells (Dijkstra) + stay-put.
2. For each candidate, find best attack available from it (explicit if a visible enemy is in
   range; else assume aggressive auto-attack).
3. Score: `1.0·expectedDamageDealt − 0.8·expectedDamageTaken (sum of visible enemies that can
   hit the cell next round, using real combat math) + 0.3·terrainArmorBonus + 0.15·advance
   (reduction in BFS distance to nearest visible enemy, or to enemy anchor cell if none
   visible) + 0.5·focusFire (target already attacked by an earlier-planned ally this round)`.
4. Stance: `defensive` if no attack chosen and expectedDamageTaken > 0; else `aggressive`.
5. Tie-breaks via FNV hash, never `Math.random`.

Weights in `data/ai.json`. Acceptance bar: the AI must reliably beat a do-nothing opponent
(test §13.6) and put up a credible fight; it does not need to be strong.

## 9. UI / UX — mobile-first, three feedback layers

Philosophy: **the board is the screen.** Portrait phone first; desktop just gets more board.
Only what needs to be visible is visible; everything else is a tap away in a sheet or modal.
The chronic v1 failure to fix: at any moment the player must see *what can be decided, what is
about to happen, and what just happened*.

### 9.1 Layout

```
┌─────────────────────────────┐
│ ◆●▲ BRUMACHLYS II ·a3f2  R3 │  ← slim top bar: version badge (§10.3), round, phase chip
├─────────────────────────────┤
│                             │
│        BOARD (SVG)          │  ← full remaining viewport; pinch-zoom + pan
│                             │
│   (selection overlays,      │
│    ghost orders, fog mist)  │
│                             │
├─────────────────────────────┤
│ [unit chips ○○●●●]  COMMIT  │  ← bottom dock: per-unit order status + commit
└─────────────────────────────┘
```

### 9.2 Layer 1 — what CAN be decided

- Tap own unit → it lifts slightly; reachable cells tint by remaining budget; visible enemies
  in range get a pulsing ring; the unit's vision edge shows as a faint contour.
- Bottom dock: one chip per own unit (its glyph), filled = has orders, hollow = not yet.
  Tap a chip → select + center that unit. Commit button shows `5/8`.
- Stance: small 3-icon popover attached to the selected unit (sword / shield / crossed-out).

### 9.3 Layer 2 — what is ABOUT to happen

- Every queued move renders as a persistent ghost trail (dotted, faction color) ending in a
  translucent ghost of the unit at the destination.
- Queued attacks render as a thin arc from unit to target with a small sword marker.
- Tap any ghost → bottom sheet: order details, [edit] [remove].
- Convergence warning: two friendly moves ending on the same cell → both ghosts flash amber.

### 9.4 Layer 3 — what HAPPENED

- On commit, resolution auto-plays. An **initiative timeline strip** slides up over the dock:
  small unit glyphs in slot order, the active slot highlighted as events play.
- Moves animate along their resolved path (~250 ms/cell). Attacks: flash arc + floating damage
  number; brawls: clash burst on the cell.
- **Tap any floating number or timeline slot → breakdown modal:** `A + Ta − D − Td + B → p →
  damage`, each term labeled, gang-up contributions itemized by class. The math is never
  invisible (v1 lesson).
- Speed control: 1× / 2× / skip. After playback: round summary sheet (kills, damage, fizzles).

### 9.5 Info on demand

Long-press any cell → info sheet: terrain stats; unit card (stats, current count, stance) if
occupied. No persistent side panels anywhere.

### 9.6 Shell screens

Start screen (donor map picker with generated previews, seed field, Battle button), win/draw
banner modal → New Battle (§4.3). That's all of v1's chrome.

## 10. Visual design — Townscaper pastel

### 10.1 Palette

Background `#F2EEE3` (warm paper). Cells get a 1.5px stroke of their fill darkened ~12%, and
corner-rounding (Catmull-Rom or quadratic smoothing of the polygon ring, radius clamped to
shortest-edge/3) — soft pebble look, no sharp shards.

| element | fill |
|---|---|
| plains | `#CBE3A8` |
| woods | `#9CCB9F` + 2–3 darker dot-clusters `#5E9B72` |
| mountains | `#CFC8BC` + ridge stroke `#A89F90` |
| swamp | `#B7C4A0` + horizontal dash texture `#93A37F` |
| water | `#A8D4E8` |
| base | `#E8D7A8`, tinted toward owner faction |
| faction A (player) | coral `#E8806B` |
| faction B (AI) | indigo `#7B8BD9` |
| fog mist | white wash `rgba(255,255,255,0.55)` over non-visible cells + slight desaturation |

Texture: one subtle SVG grain/noise filter over the whole board, opacity ≤ 0.05.

### 10.2 Units

Token = rounded squircle in faction color, white minimalist glyph, count numeral in a small
corner pip, stance shown as the token's stroke style (aggressive: solid; defensive: double;
hold-fire: dashed). Eight glyphs (sniper crosshair, humvee wedge, ranger chevron, infantry
dot-pair, grenadier triangle, tank rectangle, artillery arc, heavy tank double-rectangle) —
simple enough to read at 24 px.

### 10.3 Version badge (cache-busting integration)

Top-left of the top bar: the cache-bust **three-shape glyph** (◆●▲ — shapes/colors derived
deterministically from the current bust token) followed by the project name and the short
token in small type: `◆●▲ BRUMACHLYS II ·a3f2`. One glance at a phone tells you which build
you're on. Implemented with the `/cache-busting` toolkit (shape favicon + badge); the token
re-bumps on every deploy.

### 10.4 Skin-swap contract

All board/unit drawing goes through `src/ui/skin/` — `CellRenderer`, `UnitRenderer`,
`EffectRenderer` keyed by terrain/unit-type/event. Game components never draw primitives
directly. Replacing glyphs with sprites or animated art later = new skin module, zero game
code touched.

## 11. Tech stack & module layout

Vite + React 18 + TypeScript (strict, `noUncheckedIndexedAccess`) + Zustand + Vitest.
Board rendering: **SVG** (irregular polygons, CSS transitions, DOM hit-testing; ≤250 cells is
trivial perf). No CSS framework; plain CSS, mobile-first breakpoints.

```
src/
├─ board/        # mesh kernel port: poisson, delaunay (vendored), dissolve, subdivide,
│  │             # relax, halfedge, dual, geometry.ts, generate.ts  — PURE
├─ core/         # rng, types, orders, combat/ (ResolutionModel, weewar.ts, gangup.ts),
│  │             # pathing, fog, resolver  — PURE
├─ ai/           # FactionView builder, OrderPlanner, planner-greedy.ts  — PURE
├─ io/           # weewar-xml.ts (donor parser), data-loader.ts
├─ state/        # Zustand store: game slice, ui slice, replay slice
├─ ui/           # components + skin/
data/            # units.json, terrain.json, ai.json, scenarios.json, maps/*.xml
test/            # vitest; mirrors src structure
```

Purity guard: an ESLint `no-restricted-imports` / `no-restricted-globals` rule (or a small
custom check script run in CI) enforcing §0's determinism rules on `board|core|ai`.

## 12. Build phases

Each phase: implement → tests green → visual/manual verification where applicable → commit
→ deban sync. Don't start phase N+1 with phase N red.

- **P0 — Scaffold & ship pipeline.** Vite app boots; Vitest runs; ESLint purity rule;
  GitHub repo `kai-denrei/brumachlys-ii`; Actions workflow deploying `dist/` to Pages
  (vite `base: '/brumachlys-ii/'`); cache-busting toolkit installed; placeholder page with
  version badge LIVE on Pages. *Shipping pipeline proves out first, not last.*
- **P1 — Board kernel.** Port mesh pipeline + halfedge + dual from oskar-procedure to TS.
  `generateBoard` w/o donor (uniform plains disc) deterministic. Tests: determinism, watertight
  adjacency (symmetric neighbor lists), all cells ≥3 neighbors, geometry helpers (§13.2).
- **P2 — Donor pipeline.** Port weewar-xml parser (+ its v1 tests). Terrain assignment,
  silhouette deletion, connectivity guard, force placement. Test against bundled donors.
  Script `scripts/scan-donors.mjs` to shortlist candidates from the stash; bundle 5.
- **P3 — Core rules.** rng (port), combat weewar model (port; vectors §5.4), angle gang-up
  (§13.3), Dijkstra pathing on graph, fog. All pure, fully tested.
- **P4 — Resolver.** Phases A / A.5 / B, conflicts, charges, brawls, counter, fizzle, kill
  removal, event log. Determinism + order-shuffle invariance tests (§13.5).
- **P5 — AI.** FactionView, greedy planner, fairness + determinism + beats-do-nothing tests.
- **P6 — Board render.** Start screen → generated board on screen: pastel skin, rounded cells,
  textures, units, fog mist, pinch-zoom/pan. Verify on phone-sized viewport.
- **P7 — Order entry.** Layer-1 and Layer-2 feedback complete (§9.2–9.3). Playwright or manual
  script: queue/edit/remove all order kinds.

### 13.1 Port from v1 wholesale

### 13.2 Board kernel
- `generateBoard(donor, seed=7, n)` twice → deep-equal boards (ids, polygons, neighbors).
- Adjacency symmetric: `b ∈ neighbors(a) ⇔ a ∈ neighbors(b)`; no self-neighbors.
- `angleAt`: pivot (0,0), a at (1,0): b at (−1,0.01) → ≈179° ; b at (0,1) → 90° ;
  b at (0.94,0.34) → ≈20°.
- `graphDistance`: 0 to self; 1 to any neighbor; symmetric.

### 13.3 Angle gang-up
Defender D. Attack 1 from cell at bearing 0°, adjacent, melee. Attack 2 from bearing 170° →
prior classified opposite, B=3. Attack 3 from bearing 90° → priors: one at 90° (flanking +2),
one at 80° (flanking +2) → B=4. A prior RANGED attack (graphDistance 3) → contributes +1
regardless of bearing. Counter-attacks never appended to the accumulator.

### 13.4 Brawl
Tank(10) charges Infantry(10) on plains. Exchange 1: tank deals 5 (A5−D6→p.45×10), infantry
deals 4 (A3−D5→p.40×10) → 6 vs 5. Repeat exchanges until one side is 0; assert the sequence
and survivor deterministically.
